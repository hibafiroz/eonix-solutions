> Data Modeling:

It means designing:
- how data is stored in collections,
- how documents are structured and
- how relationships btw documents are managed.


> System Storage Basics:
MongoDB stores data on disk, but uses RAM for fast access.

a. RAM —> Random Access Memory.
> It is Temporary storage
> Stores data only while the computer is ON.
> It is very fast So, CPU uses this for run programs.
> It has Small capacity : 8 GB, 16 GB, 32 GB

2 Disk:
> It is Permanent storage
> Data remains even after we shutdown the system
> But it is Slower than RAM
> It has Large capacity: 256 GB, 1 TB, 2 TB
> It is classified into two types:
1. HDD > Hard Disk Drive - older, slower, uses magnetic plates
2. SSD > Solid State Drive - modern, fast, uses electronic memory blocks
Examples stored on disk: Photos, Videos, Apps, System files, MongoDB databases


> How MongoDB Stores Data:

MongoDB stores data on disk by dividing the disk into small fixed-size storage units called pages 
and documents are written and read inside these pages.


> WiredTiger:

WiredTiger is MongoDB’s default storage engine, and it controls how data is stored and managed on disk.

Key Features:
It stores data in compressed format.
It organizes documents using B-Trees.
It uses document-level locking, so only the modified document is locked.
It includes crash recovery mechanisms to protect data and restore it after failures.


> Document Embedding & Document References:

1. Document Embedding:
means storing related data directly inside the parent document using nested objects or arrays.
It's fast bcz mongoDB can return everything in just one query, without looking up other collections.

When to Use Embedding:
a. If there are only a small number of related items.
b. As document size limit is 16MB, so the embedded data not grow too large.

2. Document References:
Document referencing means storing related data in separate collections and linking them using IDs 
instead of embedding everything inside one document.

When to Use References:
a. Many-to-Many Relationships: Both sides can grow large, so embedding is not possible.
b. Frequently Changing Data.
c. If the child is not always read along with the parent, it deserves its own collection.
